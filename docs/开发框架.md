# 大豆杂交育种管理系统 (SoyCross) - V2 开发框架

## 1. 核心变革

从 **“静态大表 (N x N)”** 转向 **“事件驱动 (Event-Based)”** 架构。

| 特性               | 旧版 (Legacy)                     | 新版 (V2)                              | 优势                           |
| :----------------- | :-------------------------------- | :------------------------------------- | :----------------------------- |
| **存储方式** | 预生成所有可能的组合 ($N^2$ 行) | 仅存储实际发生的杂交事件               | 数据库体积减小 99%，查询极快   |
| **矩阵视图** | 物理存在的 RDS/Excel 文件         | **虚拟矩阵 (Virtual Matrix)**    | 新增亲本无需重构文件，实时渲染 |
| **唯一性**   | 靠人工检查                        | **逻辑强制** (Check-Before-Plan) | 系统级防止重复配置             |

## 2. 关键概念解释

### 2.1 虚拟矩阵 (Virtual Matrix)

虚拟矩阵不是一个物理文件，而是一种**动态视图**。

1. **行/列生成**：系统读取所有“活跃亲本”列表，在内存中构建网格。
2. **内容填充**：系统查询数据库 `SELECT * FROM crosses`。
3. **渲染**：
   * 如果数据库中有记录 (A, B)，则在网格 (Row A, Col B) 显示状态（如“2023夏”）。
   * 如果数据库无记录，则网格显示为空白。
4. **优点**：
   * 添加亲本 P100 时，无需在数据库中插入 200 行空记录，只需在亲本表中加 1 行。矩阵视图会自动多出一行一列。

### 2.2 唯一性约束 (Business Rule)

**规则**：已配置过的组合（无论成功失败），不再进行第二次配置。
**实现**：

* **数据库层**：允许存储历史记录（以备查询）。
* **应用层 (Planning)**：在生成新杂交计划时，执行严格检查：
  ```sql
  -- 伪代码
  IF EXISTS (SELECT 1 FROM crosses WHERE female_id = 'A' AND male_id = 'B') 
  THEN SKIP
  ```

### 3.2 数据模型 (SQLite)

#### 3.2.1 `parents` (亲本表)

| 字段           | 类型 | 说明                                                                                                  |
| :------------- | :--- | :---------------------------------------------------------------------------------------------------- |
| `id`         | TEXT | 主键 (P0001)                                                                                          |
| `name`       | TEXT | 唯一名称（已建立索引 idx_parents_name）                                                               |
| 若干性状列     | TEXT | 例如：`审定编号`、`来源`、`适宜区域`、`特征特性`、`特点`、`转基因` 等（当前库为宽表形式） |
| `active`     | INT  | 1=启用, 0=停用（当前库字段类型为 REAL，建议迁移为 INT）                                               |
| `created_at` | TEXT | 创建时间                                                                                              |
| `updated_at` | TEXT | 更新时间                                                                                              |

注：如需更灵活的性状扩展，可在后续版本引入 `traits` JSON 列，但现阶段与数据库保持一致采用宽表结构。

#### 3.2.2 `crosses` (杂交记录表)

*记录实际发生的杂交事件*

| 字段              | 类型    | 说明                                                                                  |
| :---------------- | :------ | :------------------------------------------------------------------------------------ |
| `id`            | TEXT    | 主键 (P001_P002)，仅由母本与父本组成，不包含批次/content                              |
| `female_id`     | TEXT    | 母本 ID（建议建立外键至 parents.id）                                                  |
| `male_id`       | TEXT    | 父本 ID（建议建立外键至 parents.id）                                                  |
| `batch`         | TEXT    | 批次号 (2024S)                                                                        |
| `name`          | TEXT    | 组合名（必填；用于账本与展示），默认可用 `female_id-male_id` 生成，再由业务规则覆盖 |
| `seed_count`    | INTEGER | 杂交获得的粒数（授粉后记录；默认 0/NA）                                               |
| `status`        | TEXT    | planned, success, failed                                                              |
| `is_reciprocal` | INT     | 1=反交                                                                                |
| `created_at`    | TEXT    | 创建时间                                                                              |
| `updated_at`    | TEXT    | 更新时间                                                                              |

索引：`idx_crosses_female`、`idx_crosses_male`、`idx_crosses_batch`、`idx_crosses_name`（已在迁移脚本中创建）。

唯一性建议：为保证同一批次内不重复，建议新增联合唯一约束或业务检查 `(female_id, male_id, batch)`；同时允许不同批次保留历史。

规划约束：配置杂交组合时必须生成并写入 `name`；若数据库存在同一 `(female_id, male_id)` 且同批次的记录，则禁止再次配置（Check-Before-Plan）。

## 4. 迁移策略 (Legacy -> V2)

1. **清洗**：读取旧 `cross_table.rds`。
2. **过滤**：丢弃所有 `content` 为空的行。
3. **转换**：
   * 提取 `batch` (从 content 文本)。
   * 生成 ID：`母_父_批次` (保留历史痕迹)。
4. **入库**：写入 SQLite。

## 5. 核心功能模块开发计划

### 5.1 杂交计划模块 (`mod_cross.R`)

* **输入**：目标批次名，母本列表，父本列表。
* **逻辑**：
  1. 遍历所有 (Mother, Father) 候选对。
  2. **Check**: `db$has_cross(m, f)`?
     * Yes -> 记录为“跳过 (已存在)”。
     * No -> 加入待创建列表。
  3. **Action**: 批量插入 `crosses` 表。
* **输出**：本次计划摘要 + 数据库更新。

### 5.2 虚拟矩阵视图 (`mod_matrix.R`)

* 使用 `rhandsontable` 或 `DT`。
* 动态加载：根据 `parents` 表的 `active` 字段生成框架。
* 异步填充：后台查询 `crosses` 填入单元格。

### 5.3 统计分析

- 原则：所有统计直接用 SQL 聚合与过滤；避免把整表拉到 R 端再处理
- 数据源：crosses（事件表）、parents（亲本属性表）
- 性能策略：为常用字段建立索引（female_id、male_id、batch、status、created_at），查询时使用 WHERE/ORDER BY、LIMIT

示例与约定：
- 批次与状态分布：
  ```sql
  SELECT batch, status, COUNT(*) AS n
  FROM crosses
  WHERE batch IS NOT NULL
  GROUP BY batch, status
  ORDER BY batch;
  ```
- 亲本使用频次（母/父分开与合并）：
  ```sql
  -- 母本频次
  SELECT female_id AS parent_id, COUNT(*) AS n
  FROM crosses GROUP BY female_id ORDER BY n DESC;
  -- 父本频次
  SELECT male_id   AS parent_id, COUNT(*) AS n
  FROM crosses GROUP BY male_id   ORDER BY n DESC;
  -- 总体频次（合并）
  SELECT parent_id, SUM(n) AS n FROM (
    SELECT female_id AS parent_id, COUNT(*) AS n FROM crosses GROUP BY female_id
    UNION ALL
    SELECT male_id   AS parent_id, COUNT(*) AS n FROM crosses GROUP BY male_id
  ) t GROUP BY parent_id ORDER BY n DESC;
  ```
- 反交配对诊断（同一对亲本的正交/反交是否同时存在）：
  ```sql
  SELECT a.female_id, a.male_id,
         COUNT(*) AS n_forward,
         (SELECT COUNT(*) FROM crosses b
          WHERE b.female_id = a.male_id AND b.male_id = a.female_id) AS n_reciprocal
  FROM crosses a
  GROUP BY a.female_id, a.male_id;
  ```
- 粒数分布（基本统计）：
  ```sql
  SELECT batch,
         COUNT(seed_count) AS n,
         AVG(seed_count)   AS avg_seed,
         MIN(seed_count)   AS min_seed,
         MAX(seed_count)   AS max_seed
  FROM crosses
  GROUP BY batch;
  ```
- 矩阵热力图数据（female × male 计数，含零补齐）：
  ```sql
  WITH pairs AS (
    SELECT f.id AS female_id, m.id AS male_id
    FROM parents f CROSS JOIN parents m
    WHERE f.active = 1 AND m.active = 1
  ), agg AS (
    SELECT female_id, male_id, COUNT(*) AS n
    FROM crosses GROUP BY female_id, male_id
  )
  SELECT p.female_id, p.male_id, COALESCE(a.n, 0) AS n
  FROM pairs p LEFT JOIN agg a
  ON p.female_id = a.female_id AND p.male_id = a.male_id;
  ```
- 性状影响的简单分析（示例：生长习性与成功率）：
  ```sql
  SELECT pr.growth_habit,
         SUM(CASE WHEN cr.status = 'success' THEN 1 ELSE 0 END) * 1.0 / COUNT(*) AS success_rate
  FROM crosses cr
  JOIN parents pr ON pr.id = cr.female_id
  WHERE pr.growth_habit IS NOT NULL
  GROUP BY pr.growth_habit
  ORDER BY success_rate DESC;
  ```

输出约定：统一返回整洁表（tibble）；绘图或交互展示在上层 UI/脚本实现。功能实现细节见 5.5 模块接口。
### 5.4 田间账本模块 (mod_field.R)

- 职责：生成田间种植计划与账本、导入导出 Excel/CSV、同步 crosses 状态
- 依赖：DBI/RSQLite（必需）；读写 Excel/CSV 通过 utils_io 封装
- 主要功能：
  - field_plan_create(batch, layout_opts): 从 crosses 选出指定批次的记录，生成布局数据
    - 输入：batch，布局参数（床号/行列/重复数/起始编号等）
    - 输出：tibble，列：cross_id, female_id, male_id, name, plot_code, replicate, bed_no, row_no, col_no, planting_date
  - field_book_export(plan_tbl, out_path): 调用 utils_io 写出账本（xlsx/csv），包含“组合表”“布局表”“标签表”
  - field_import_book(in_path): 读取人工编辑后的账本并校验，返回标准化数据框
  - field_update_status(plan_tbl, status = "planted"): 批量更新 crosses.status 与 updated_at；可记录 planting_date
  - field_labels(plan_tbl): 生成田间标签数据（plot_code、name、female_id、male_id、batch）
- 关键约束：
  - 仅使用数据库中的 crosses/parents 信息，不额外创建持久化表；衍生数据（布局）以导出文件或临时视图形式存在
  - 批次内布局唯一；plot_code 由 batch+顺序生成，避免重复

### 5.5 统计分析模块 (mod_analysis.R)

- 职责：提供杂交事件与亲本使用的基础统计与诊断
- 依赖：DBI/RSQLite；可选 ggplot2/DT 仅用于展示（不作为硬依赖）
- 主要功能：
  - cross_stats_by_batch(): 各批次数量、成功/失败占比
    - SQL 示例：SELECT batch, COUNT(*) AS n FROM crosses GROUP BY batch
  - parent_usage(top_n = 20): 亲本使用频次（母/父分开与合并）
    - SQL 示例：SELECT female_id AS parent_id, COUNT(*) FROM crosses GROUP BY female_id
  - seed_count_summary(batch = NULL): 粒数的分布与均值/中位数/极值
  - reciprocal_pairs(): 统计正交/反交是否同时存在及其结果差异
  - matrix_heatmap_data(): 生成 female × male 的计数矩阵数据，用于热力图
  - trait_effects_simple(): parents 结构化性状与结果的简单相关诊断（如相关系数、分组均值），不做因果推断
- 输出形式：统一返回 tibble；绘图或表格展示交由上层 UI

### 5.6 I/O 辅助模块 (utils_io.R)

- 职责：统一文件读写、路径与备份、导出模板
- 依赖：尽量优先使用 base R；如检测到 readxl/openxlsx/writexl 则使用以提升体验
- 主要功能：
  - normalize_path(p): 统一 Windows/Unix 路径分隔与绝对化
  - backup_db(db_path, dest_dir = dirname(db_path)): 备份 SQLite，返回备份文件路径（含时间戳）
  - write_table(df_list, out_path, format = c("xlsx","csv","rds")): 将一个或多个数据框写出为指定格式
  - read_table(in_path): 自动识别扩展名并读取为数据框或列表
  - export_cross_matrix(con, batch = NULL, out_path): 从 crosses 生成矩阵视图并导出
  - import_cross_matrix(in_path): 读取历史二维表并标准化为长表（female_id, male_id, content/batch）
- 安全策略：所有写操作默认不覆盖；若需覆盖，必须显式 overwrite = TRUE 并先备份

### 7.4 迁移注意事项

⚠️ **重要提示**：

1. **迁移前备份**：务必备份 `data/` 目录全部内容
2. **字段名变更**：
   - RDS 系统使用中文字段名（如 `名称`、`蛋白`）
   - SQLite 使用英文字段名（`name`、`protein`）
3. **content 字段**：RDS 系统的 `content` 字段对应 V2 的 `batch` 字段
4. **反交标记**：
   - RDS 系统通过 `content` 包含 "反交" 判断
   - V2 使用独立的 `is_reciprocal` 字段
5. **唯一性约束**：V2 在业务层强制执行（同一亲本对不能重复配置），而非数据库约束

### 7.5 兼容性说明

**完全废弃的功能**：
* `combine_records()`: 预生成 N×N 大表（V2 使用事件驱动）
* `add_parent_and_append_cross()`: 新增亲本自动补全组合（V2 需显式创建计划）

backup_before_operation <- function(db_path, operation_name) {
  # 1. 创建备份目录
  backup_dir <- file.path(dirname(db_path), "backups", operation_name)

safe_operation <- function(db_path, ...) {
  # 1. 输入验证
  if (!file.exists(db_path)) {
    stop("❌ 数据库不存在：", db_path)
  }
  
  # 2. 环境准备
  con <- NULL
  backup_path <- NULL
  
  tryCatch({
    # 3. 备份
    backup_path <- backup_before_operation(db_path, "operation_name")
    
    # 4. 连接数据库
    con <- DBI::dbConnect(RSQLite::SQLite(), db_path)
    
    # 5. 开启事务
    DBI::dbBegin(con)
    
    # 6. 执行操作
    # ... 核心逻辑 ...
    
    # 7. 提交事务
    DBI::dbCommit(con)
    
    # 8. 记录日志
    log_operation("operation", "success", details = list(...))
    
    cat("✅ 操作成功\n")
    
  }, error = function(e) {
    # 回滚事务
    if (!is.null(con)) {
      tryCatch(DBI::dbRollback(con), error = function(e2) {})
    }
    
    # 记录错误日志
    log_operation("operation", "error", 
                  details = list(error = e$message),
                  status = "error")
    
    # 输出错误信息
    cat("❌ 操作失败：", e$message, "\n")
    if (!is.null(backup_path)) {
      cat("💾 可从备份恢复：", backup_path, "\n")
    }
    
    stop(e)
    
  }, finally = {
    # 清理资源
    if (!is.null(con)) {
      DBI::dbDisconnect(con)
    }
  })
}

- 删（DELETE / 软删除）
  
  ```r
  # 软删除：active 置 0，保留引用
  dbExecute(con, 
    "UPDATE parents SET active = 0, updated_at = ? WHERE id = ?",
    params = list(format(Sys.time(), "%Y-%m-%d %H:%M:%S"), "P1001"))
  
  # 物理删除前检查引用
  ref_n <- dbGetQuery(con,
    "SELECT 
       (SELECT COUNT(*) FROM crosses WHERE female_id = ?) +
       (SELECT COUNT(*) FROM crosses WHERE male_id   = ?) AS n",
    params = list("P1001", "P1001"))$n
  if (ref_n == 0) {
    dbExecute(con, "DELETE FROM parents WHERE id = ?", params = list("P1001"))
  } else {
    stop(sprintf("❌ 禁止删除：存在 %d 条 crosses 引用", ref_n))
  }
  ```

## 9. 常见问题与解决方案

### 9.1 数据库锁定问题

**问题**：`database is locked` 错误

**解决方案**：
```r
# 1. 检查是否有未关闭的连接
DBI::dbListConnections(RSQLite::SQLite())

# 2. 强制关闭所有连接
purrr::walk(DBI::dbListConnections(RSQLite::SQLite()), DBI::dbDisconnect)

# 3. 设置超时时间
con <- DBI::dbConnect(RSQLite::SQLite(), "data/db/soy_cross.db")
DBI::dbExecute(con, "PRAGMA busy_timeout = 5000")  # 5秒超时
```

### 9.2 大批量插入性能问题

**问题**：插入上千条记录很慢

**解决方案**：
```r
# 使用事务批量插入
con <- DBI::dbConnect(RSQLite::SQLite(), "data/db/soy_cross.db")
DBI::dbBegin(con)

tryCatch({
  # 批量插入（一次性操作）
  DBI::dbAppendTable(con, "crosses", large_dataframe)
  DBI::dbCommit(con)
}, error = function(e) {
  DBI::dbRollback(con)
  stop(e)
}, finally = {
  DBI::dbDisconnect(con)
})
```

### 9.3 中文字符乱码问题

**问题**：Excel 导出或数据库中文显示乱码

**解决方案**：
```r
# 1. 确保 R 脚本使用 UTF-8 编码
Sys.setlocale("LC_CTYPE", "Chinese")

# 2. 数据库连接时指定编码
con <- DBI::dbConnect(
  RSQLite::SQLite(), 
  "data/db/soy_cross.db",
  extended_types = TRUE
)

# 3. Excel 导出使用 UTF-8 BOM
openxlsx::write.xlsx(
  data, 
  "output.xlsx", 
  asTable = FALSE,
  encoding = "UTF-8"

```
